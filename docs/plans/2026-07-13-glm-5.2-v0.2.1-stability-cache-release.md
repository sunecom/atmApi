# atmApi GLM-5.2 V0.2.1 稳定性、缓存与可追溯发布 Implementation Plan

> **For Claude/Codex/艾隆:** REQUIRED SUB-SKILL: Use `executing-plans` to implement this plan task-by-task. 未经建国确认，不得进入生产部署步骤。

**Goal:** 在保持“请求 `glm-5.2` 就只使用真实 GLM-5.2”的前提下，解决 reasoning 空可见输出、伪流式、错误熔断和源码/二进制漂移，并建立可验证的 OpenRouter 缓存、省 Token 与可回滚发布链路。

**Architecture:** GLM-5.2 请求通过独立 `internal/glmoptimizer` 管线完成模型锁定、请求规范化、reasoning 预算、上下文预算、缓存键生成和终态判定，再交给只选择 `model_group=glm-5.2` 的路由策略。OpenRouter 作为一个“内部已聚合”的上游渠道，atmApi 不复刻其 Provider 选择算法，只使用稳定 `session_id`、规范请求和必要的二级 failover。流式响应采用“逐事件原样转发 + 旁路状态解析”，一旦向客户端发送首字节即禁止透明重试。

**Tech Stack:** Go 1.22.5、Gin、GORM、MariaDB 10.5、OpenAI Chat Completions/SSE、OpenRouter、systemd、Git、Go 标准测试与 `httptest`。

## 1. 结论先行

艾隆提出的三个紧急项方向正确，但不能继续沿用现有止血实现：

1. `max_tokens` 是 reasoning 与可见答案共享的输出预算。缩短输入上下文只能降低输入费用，不能直接给可见答案“腾出”输出额度。
2. `content == ""` 不是充分的失败条件；完整 `tool_calls` 或 `refusal` 都可能是合法终态，只有 reasoning 而没有任何用户可消费结果则属于业务失败。
3. SSE 不能先全量缓冲再一次性写回。正确实现是逐事件转发，同时旁路解析状态；首字节发出后不能透明切换渠道重试。
4. 400/422 等请求错误不应污染渠道熔断；401/403、402、429、5xx、网络错误必须按不同故障域处理。
5. 本地响应缓存只能缓存严格等价、无工具副作用、显式确定性的非流式请求；上游 Prompt Cache 才是长对话省钱主力。
6. 源码、构建产物和运行进程必须绑定同一 Git SHA，由 systemd 统一管理，禁止再次手工复制未知来源二进制。

## 2. 目标、非目标与验收原则

### 2.1 本版目标

- 锁定真实 GLM-5.2，不跨模型、不降级到 Qwen/DeepSeek/GLM-4.x。
- 对 reasoning、正文、工具调用、拒答和 usage 做结构化解析。
- 恢复真正的 SSE 首包转发，避免 reasoning 期间长时间无输出。
- 对非流式空可见结果执行一次有边界的同模型渠道重试。
- 对流式末端空可见结果返回明确错误事件，不伪装成功。
- 建立精确错误分类、半开探测和最多两渠道的 failover。
- 通过规范 JSON、稳定前缀、稳定 `session_id` 和单飞合并提升 OpenRouter 缓存效率。
- 记录真实 cached/reasoning token 与上游成本，不再用固定 10% 缓存折扣估算所有渠道。
- 建立可复现构建、systemd 切换和一分钟内回滚能力。

### 2.2 本版明确不做

- 不做 `glm-5.2-smart` 跨模型路由。
- 不自动改写用户问题或用语义相似缓存返回答案。
- 不在生产默认开启 LLM 自动摘要；先做 shadow 观测和离线对照。
- 不承诺 OpenRouter 当前促销价长期有效。
- 不把多个同源转售渠道宣传成独立故障域。
- 不顺带重构支付、淘宝发货或 `deepseek-a4` 路径。

### 2.3 硬性验收原则

任一项不满足即不得生产灰度：

- 任何 GLM-5.2 请求都没有跨模型实际调用记录。
- 流式首事件实时到达客户端，不等待上游 `[DONE]`。
- tool call 被正确识别为成功，不能被判定为空响应。
- 400/422 不计入熔断；429/5xx/网络错误按配置计入。
- 请求关键字段任一改变时，本地缓存必须 miss。
- 发布二进制可从记录的 Git SHA 复现且哈希一致。
- systemd 能自动拉起，回滚步骤已实际演练。

## 3. 三种方案与最终取舍

| 方案 | 做法 | 优点 | 缺点 | 结论 |
|---|---|---|---|---|
| A. 独立 GLM 管线 | 新建 `internal/glmoptimizer`，旧路径只做入口分流 | 隔离风险、可测试、不影响 A4 | 有少量重复适配代码 | 采用 |
| B. 继续在 `routes.go/router.go` 打补丁 | 在现有大函数中增加判断 | 表面改动少 | 顺序依赖严重，缓存、压缩、路由继续互相污染 | 拒绝 |
| C. 新建独立微服务 | GLM-5.2 单独端口和进程 | 隔离最彻底 | 首版运维和数据一致性成本过高 | 暂缓 |

最终采用 A。它保留现有单体部署，但在代码内形成清晰边界；如果未来 GLM 业务量足够大，可再把该包平移为独立服务。

## 4. 高层架构与数据流

```text
Client
  -> auth / plan / rate limit
  -> GLM model lock
  -> request parser + canonicalizer
  -> reasoning/output budget guard
  -> tool-safe context budget
  -> exact-cache eligibility + singleflight
  -> GLMRoutePolicy
       -> OpenRouter z-ai/glm-5.2（内部 Provider 聚合）
       -> 其他已准入的真实 GLM-5.2 渠道（同模型二级容灾）
  -> non-stream terminal classifier
     or true SSE relay + sidecar observer
  -> usage / cost / cache / failure telemetry
```

### OpenRouter 的边界

- 对 atmApi 而言，`z-ai/glm-5.2` 是一个聚合模型入口，不是单一物理 Provider。
- 默认顺应 OpenRouter 的 Provider 选择与 fallback；不同时硬编码 `provider.order` 和粘性路由。
- 有可靠会话标识时写入稳定、不可逆的 `session_id`，提高同会话落到同一 Provider 和 Prompt Cache 的概率。
- `session_id` 使用 HMAC 派生，不传原始 atmApi Key、用户 ID 或隐私标识。
- OpenRouter 整体出现可重试故障时，atmApi 才切换到第二个真实 GLM-5.2 渠道。
- 单次请求最多尝试两个 atmApi 渠道，禁止多层重试风暴。

## 5. ADR：关键架构决策

### ADR-001：GLM-5.2 使用独立优化管线

**Status:** Proposed

**Context:** 当前 `chatCompletions` 同时承担鉴权、路由、缓存、工具压缩、上下文摘要、行为注入、SSE 与 usage 解析；处理顺序已经导致缓存键早于最终请求生成、模型锁被任务路由覆盖等问题。

**Decision:** 新建 `internal/glmoptimizer`，只在请求模型或套餐组属于 GLM-5.2 时进入；旧 A4 路径不改语义。

**Consequences:** 风险可隔离、可单测；代价是首版会保留少量协议适配重复代码。

**Alternatives:** 继续修改大函数、拆独立微服务；分别因回归风险和运维成本被拒绝。

### ADR-002：真流式旁路观察，首字节后不重试

**Status:** Proposed

**Context:** 全量缓冲虽能在末端判断空正文，但破坏 SSE，增加内存并让用户等待全部 reasoning。

**Decision:** 每个完整 SSE event 到达后立即写回并 flush，同时解析副本更新终态。上游首字节前可以按错误分类切换；首字节后绝不透明重试。

**Consequences:** 保住首包时延和协议语义；流中断只能显式失败，不能无痕换渠道。

### ADR-003：终态由可消费产物判定

**Status:** Proposed

**Decision:** 成功条件为至少一个非空正文、完整 tool call 或非空 refusal。`reasoning`/`reasoning_details` 单独存在不算用户可消费成功。流式在 `[DONE]` 前完成判定；不合格时先写标准错误 event，再写 `[DONE]`。

**Consequences:** 不误伤工具调用；reasoning 耗尽预算不再伪装成功。

### ADR-004：缓存键以“最终实际发送请求”为准

**Status:** Proposed

**Decision:** 所有模型锁定、预算调整、上下文处理和确定性注入完成后，才对最终请求做规范序列化；同一份规范字节既用于哈希也用于发送上游。

**Consequences:** 避免键与实际请求不一致；改变工具、reasoning、response format 等字段必然 miss。

### ADR-005：发布以 Git SHA 为唯一来源

**Status:** Proposed

**Decision:** 只从 clean worktree 构建；产物目录包含 Git SHA 和 SHA256；systemd 只运行 `current` 指向的版本目录；禁止手工 `cp` 到部署目录。

**Consequences:** 可审计、可回滚；首次切换需要安排短维护窗口清理裸进程。

## 6. Reasoning 与可见输出预算

### 6.1 必须纠正的认知

- 输入 Token 与输出预算是两套限制。
- 缩短 54K 输入可以省输入费用，但不会改变 `max_tokens` 在 reasoning 与正文之间的分配。
- OpenRouter 对 reasoning effort 的通用映射可用于压测规划，但不是 GLM-5.2 的固定服务承诺；必须用真实账单和响应验证。

### 6.2 当前套餐硬上限的规划估算

以下只用于 Phase 0 设计测试矩阵：按 high 约 80%、xhigh 约 95% 输出预算用于 reasoning 粗估。

| 套餐 | `max_tokens` 硬上限 | high 粗估可见余额 | xhigh 粗估可见余额 | V0.2.1 结论 |
|---|---:|---:|---:|---|
| basic | 8,192 | 约 1,638 | 约 410 | 不保证 xhigh 可见答案 |
| standard | 16,384 | 约 3,277 | 约 819 | xhigh 仅实验 |
| pro | 32,768 | 约 6,554 | 约 1,638 | xhigh 若要求 2K 正文仍可能不足 |

### 6.3 预算守卫规则

1. 解析 `max_tokens`、`reasoning.enabled`、`reasoning.effort`、`reasoning.max_tokens` 和套餐硬上限。
2. 未开启 reasoning 时保持原样，不由网关强制开启；`reasoning.enabled=true` 但未指定 effort 时规范为 `high`。
3. V0.2.1 首版对显式 `xhigh` 返回结构化 400，不静默降为 high；只有 Phase 0A 数据和建国确认后才允许开放。
4. 若 OpenRouter+GLM 实测确认支持 `reasoning.max_tokens`，优先执行：
   `reasoning.max_tokens <= effective_max_tokens - min_visible_tokens`。
5. 若只提供 effort，则用已校准系数做准入估算；组合无法满足可见输出保留量时，在请求发出前返回 400 和明确建议。
6. 不静默抬高套餐上限；high 的 80% 仅是规划估算，不构成可见 Token 保证。
7. 非流式出现 reasoning-only 终态时，可换一个真实 GLM-5.2 渠道重试一次；流式首字节后不重试。
8. 所有预算调整都写入结构化日志，包含 requested/effective max、effort、reasoning token、visible token 和结束原因。

### 6.4 Phase 0 实测矩阵

- 历史消息数：50、100、200、300/约 54K Token。
- `max_tokens`：8K、16K、32K，以及仅测试环境下的 48K/64K。
- reasoning：disabled、enabled 默认、high、xhigh、显式 reasoning.max_tokens。
- 响应：普通正文、代码长答、tool call、拒答。
- 记录：TTFT、总耗时、prompt/cached/reasoning/completion token、正文 token、finish_reason、上游费用和 Provider（若可见）。
- 只有矩阵完成后，才最终决定 basic/standard/pro 的默认 effort 与可见输出保留量。

## 7. 终态分类器

### 7.1 非流式状态

| 状态 | 判定 | 处理 |
|---|---|---|
| `success_content` | 任一 choice 的 `message.content` 有非空文本 | 成功 |
| `success_tool_call` | 存在结构完整的 tool call | 成功 |
| `success_refusal` | refusal 非空 | 成功 |
| `failure_reasoning_only` | 只有 reasoning，无上述产物 | 未向客户端写出时可同模型重试一次 |
| `failure_empty` | choices 为空或所有产物为空 | 未向客户端写出时可同模型重试一次 |
| `failure_malformed` | 200 但 JSON/协议损坏 | 计渠道协议故障并尝试下一同模型渠道 |

完整 tool call 至少要满足可识别的 call ID、function name，arguments 能在结束时形成完整字符串；流式 delta 需要按 index 聚合后判断，不能做字符串搜索。

### 7.2 流式状态机

状态字段：`bytes_forwarded`、`content_seen`、`refusal_seen`、`tool_calls_by_index`、`reasoning_seen`、`finish_reason`、`usage_seen`、`done_seen`、`parse_errors`。

处理顺序：

1. 按 SSE event 边界读取，而不是按任意 TCP chunk 判 JSON。
2. 保留注释行、多行 `data:` 和空行边界，原样转发。
3. 对 `data:` 的 JSON 副本做结构化解析并更新状态。
4. 每个 event 写回后立即 flush。
5. 拦截上游 `[DONE]`，先执行终态判断。
6. 合格则转发 `[DONE]`；不合格则写 OpenAI 兼容错误 event，再写 `[DONE]`。
7. 客户端断开时取消上游 context，释放并发槽，不再计正常成功。

## 8. 错误分类、熔断和重试

| 类别 | 典型情形 | 是否换渠道 | 是否计瞬时熔断 | 额外动作 |
|---|---|---:|---:|---|
| `client_request` | 400、422、工具链不完整、参数非法 | 否 | 否 | 原样返回可读错误 |
| `channel_auth` | 上游 401、403 | 是 | 否 | 标记配置异常、告警，等待人工修复 |
| `channel_balance` | 402、余额耗尽 | 是 | 否 | 长冷却并告警，不伪装成几分钟限流 |
| `channel_rate_limit` | 429 | 是 | 是 | 尊重 Retry-After |
| `channel_transient` | 408、5xx、连接失败、首字节前超时 | 是 | 是 | 指数退避受总 deadline 约束 |
| `channel_protocol` | 200 但 JSON/SSE 损坏 | 是 | 是 | 保留脱敏样本 |
| `semantic_empty` | 200 但 reasoning-only/完全空 | 非流式是；已流式否 | 单独计数 | 检查预算与 Provider |
| `client_cancel` | 客户端主动断开 | 否 | 否 | 取消上游、释放资源 |

熔断器按 `channel.ID` 维护，不按名称；默认阈值“60 秒内 3 次可计失败、打开 60 秒”仅作为初值。打开后只允许一个 half-open 探针；成功关闭，失败重新打开。任何成功请求不能粗暴清除全部历史，只能推动当前窗口恢复。

重试边界：

- 单请求最多两个 atmApi 渠道。
- 仅首字节前允许透明切换。
- GLM-5.2 只能切到 `model_group=glm-5.2` 且已准入的渠道。
- 禁止现有“tool_calls 错误后切 Qwen”逻辑进入 GLM 路径。
- 超时由请求总 deadline 控制，不能每个渠道各获得完整 30/90 秒。

## 9. 缓存与省 Token 设计

### 9.1 四层策略

1. **稳定前缀：**保留 system、tools 和长期约束的顺序；对象 key 规范化；不在前缀插入时间戳、随机 ID、动态统计。
2. **OpenRouter Prompt Cache：**同一会话使用稳定 HMAC `session_id`；不无故指定 `provider.order`；从 usage 读取 cached token。
3. **并发单飞：**对严格可缓存的相同请求合并正在进行的上游调用，弥补 OpenRouter 精确响应缓存不合并并发请求的限制。
4. **本地精确响应缓存：**只缓存确定性、无工具副作用的非流式完整成功结果；它是补充，不是长对话主策略。

### 9.2 规范化与缓存键

规范化必须发生在全部请求变换之后。禁止重排数组；仅稳定 JSON 对象 key、数值表示和已确认的默认字段。规范字节直接发送上游，避免“哈希内容”和“实际请求”不同。

缓存键最少覆盖：

- 租户/Token 的 HMAC scope，不保存原始 Key。
- 实际锁定模型与策略版本。
- 最终 `messages`、`tools`、`tool_choice`。
- `reasoning` 全对象。
- `temperature`、`top_p`、`max_tokens`、`stop`、`seed`、`n`。
- `response_format`、`modalities`、`stream_options` 及所有透传的行为字段。
- 最终 `session_id` 和影响 Provider 选择的 `provider` 对象。

### 9.3 本地响应缓存准入

V0.2.1 默认只有同时满足下列条件才准入：

- `stream=false`。
- 客户端显式提供 `temperature=0`；缺省 temperature 不再当作 0。
- 不含 `tools`/`tool_choice`、图片、音频或其他可能带副作用/时效性的模态。
- `n=1`，无未识别的行为字段。
- 终态为 `success_content`，并通过完整 JSON 校验。
- 未命中用户显式 `Cache-Control: no-store` 或 atmApi 禁用标记。

缓存命中仍记录套餐限流与审计记录，但上游 token/cost 为 0，并用独立字段标记 `local_response_cache_hit=true`，不能伪造上游 cached token。

### 9.4 上下文压缩的 V0.2.1 边界

- 停止把“删除历史”称为无损截断。
- GLM-5.2 首版不自动调用 DeepSeek Flash 生成摘要。
- 消息按完整 turn/tool transaction 分组，永不切断 assistant tool_calls 与对应 tool 结果。
- 字符串按 UTF-8 rune/Token 边界截断，禁止 `content[:500]` 破坏中文。
- 工具输出仅做确定性压缩：去重复日志、限制连续同类行、保留错误、文件路径、退出码和首尾证据。
- 语义摘要进入 shadow：生成候选摘要但不发给上游，对比事实保留率、成本和命中率；通过人工样本后再单独开关灰度。
- 超出安全输入预算且无法无损收敛时，返回明确错误与建议，不静默丢弃关键历史。

## 10. 用量、成本与可观测性

V0.2.1 不建设复杂不可变成本账本，但必须有可复核的最小记录：

- requested_model、actual_model、channel_id/name、policy_version。
- prompt_tokens、cached_tokens、cache_write_tokens（若上游提供）。
- reasoning_tokens、visible_output_tokens、completion_tokens。
- upstream_reported_cost 与币种（OpenRouter 返回时优先记录）。
- pricing_snapshot_id；没有上游实报成本时才按 Provider 价格快照估算。
- local_response_cache_hit、singleflight_shared、session_id_hash_prefix。
- terminal_state、finish_reason、retry_count、breaker_state。
- TTFT、总耗时、首字节前失败和流中断。

必须移除/停用 GLM 路径中固定 `cached input = input price × 10%` 的估算。已知 GLM/OpenRouter/Z.AI 缓存折扣应由渠道价格快照配置，不能写死为所有模型通用比例。

价格看板同时展示：

1. 当前实付/促销价格。
2. 常态保守价格。
3. 备用渠道切换价格。

毛利评估按三套价格分别计算；任何“约 70% 促销”只有账单证据后才能进入正式测算。

## 11. 非功能要求

| 维度 | V0.2.1 目标 |
|---|---|
| 隔离性 | GLM 改动不改变 deepseek-a4 回归结果 |
| 流式性能 | 网关新增 TTFT 开销 p95 小于 100ms（不含上游耗时） |
| 内存 | 单个 SSE 请求不缓存完整响应；内存随最大单 event 有界 |
| 可靠性 | systemd 自动重启；RTO 小于 5 分钟；发布回滚小于 1 分钟 |
| 重试 | 最多两渠道、受统一 deadline 控制 |
| 安全 | 日志不含完整 API Key、Authorization、完整用户 Prompt |
| 可审计 | 每个运行版本能映射到 Git SHA、构建时间与二进制哈希 |
| 成本 | 每笔 GLM 调用可区分未缓存输入、缓存输入、reasoning 与可见输出 |

# atmApi 项目整体方案介绍

## 一、项目定位

atmApi 不是简单复刻 One API，也不是做一个“又一个模型中转站”。

它的核心定位是：

> **One API 的国产替代 Plus**

更准确一点说：

> **一个面向中国开发者和 AI 团队的，全中文、更符合国人使用习惯、支持多渠道 fallback、后续可扩展充值/账单/管理能力的 AI 模型聚合与分发平台。**

这个项目既是产品，也是 AiToMoney 团队对外展示能力的重要载体。

---

## 二、为什么要做 atmApi

### 1. 表层原因：现有方案不够贴合国人场景

目前市场上已有 One API 这样的优秀项目，但对中国用户来说，仍有几个明显问题：

- 术语、交互、界面不够本地化
- 很多功能偏技术人员视角，非技术团队不容易直接上手
- 对“多渠道自动 fallback”这一类高可用能力支持不够顺手
- 对团队管理、配额管理、充值账单、业务运营场景支持还不够贴近国内需求

也就是说，用户真正需要的不是“再一个中转站”，而是：

> **一个更适合中国 AI 团队实际使用的模型管理后台。**

### 2. 深层原因：AiToMoney 需要自己的品牌级产品

第一性原理看，团队知名度不是靠喊口号，而是靠：

- 做出真实可用的产品
- 形成可传播的项目资产
- 让别人提到某个问题时，第一时间想到你

atmApi 承担的就是这个角色。

它的价值不只是“能转发请求”，而是：

- 对外：成为 AiToMoney 团队的代表性项目
- 对内：成为团队统一模型管理与调度底座
- 对传播：成为 GitHub、文档、演示、社群传播的抓手

所以这个项目既服务业务，也服务品牌。

## 三、项目目标

atmApi 的目标分三层。

### 第一层：短期目标（MVP）

在最短时间内做出一个 **可运行、可配置、可演示** 的最小可用产品：

- 支持多渠道接入
- 支持模型路由
- 支持自动 fallback
- 支持 Token 管理
- 支持渠道管理
- 支持简单后台界面
- 支持公网访问和演示

这一层的目标不是完美，而是证明：

> 核心链路能跑通，产品方向成立。

### 第二层：中期目标（可管理产品）

从“能演示”走向“能管理”：

- 支持用户管理
- 支持渠道编辑/删除
- 支持 Token 编辑/删除
- 支持登录与权限控制
- 支持请求日志
- 支持配额与用量统计
- 支持更完整的后台交互体验

这一层的目标是让 atmApi 真正具备产品级使用价值。

### 第三层：长期目标（运营平台）

从“管理后台”走向“业务平台”：

- 支持充值体系
- 支持账单系统
- 支持渠道成本核算
- 支持用户分层与套餐
- 支持团队协作管理
- 支持更复杂的路由和调度策略

这一层的目标是把 atmApi 从工具升级成平台。

## 四、核心产品价值

### 1. 多渠道统一接入

将多个上游模型服务统一纳入一个系统中管理，例如：

- 通义千问
- DeepSeek
- GoToken
- 未来可扩展 OpenAI / Anthropic / 其他国内外模型接口

这样做的好处是：

- 用户只面对一个入口
- 配置集中管理
- 上游切换成本低
- 有利于后续统一计费和统计

### 2. 自动 fallback

这是 atmApi 当前最核心、最有感知价值的能力之一。

逻辑很简单：

- 先走主渠道
- 主渠道失败（如 4xx/5xx/超时）
- 自动切换到备用渠道

这能直接解决一个真实问题：

> 模型服务不稳定时，业务不能跟着一起停。

高可用能力，永远是中转层最重要的价值之一。

### 3. 全中文后台与本地化体验

不是每个用户都习惯英文术语、工程师风格后台。

atmApi 从一开始就强调：

- 中文界面
- 国人熟悉的交互方式
- 更明确的字段命名
- 更接近国内团队的使用习惯

这会显著降低使用门槛。

### 4. 可扩展的业务化能力

很多开源项目停在“技术可用”，但很难走到“业务可运营”。

atmApi 一开始就预留业务化方向，包括：

- 用户配额
- Token 管理
- 渠道优先级
- 路由策略
- 后续充值与账单体系

也就是说，它不是一个只给工程师自用的脚手架，而是一个有潜力成长为业务平台的底座。

## 五、核心功能架构

当前与未来的核心功能可分为以下几个模块：

### 1. 用户管理模块

作用：
- 管理系统使用者
- 记录角色、状态、配额
- 为后续权限控制与团队管理打基础

当前状态：
- 已有基础数据模型
- 后续将补全登录、权限与界面支持

### 2. Token 管理模块

作用：
- 为调用方生成 API Token
- 绑定用户
- 管理剩余配额、过期时间、状态

价值：
- 统一调用入口
- 方便后续做用量和权限管理

### 3. 渠道管理模块

作用：
- 配置上游模型服务
- 管理 API Key、Base URL、模型列表、模型映射
- 设置优先级和权重

价值：
- 把底层模型服务变成可控资源
- 为多渠道路由和 fallback 提供基础

### 4. 模型路由模块

作用：
- 接收外部请求
- 根据模型名查找匹配渠道
- 应用模型映射
- 把请求转发到正确上游

价值：
- 统一模型调用入口
- 降低上层应用对各个供应商 API 差异的感知

### 5. fallback 模块

作用：
- 当主渠道不可用时自动切换备用渠道
- 保证请求尽可能成功返回

价值：
- 提高系统稳定性
- 降低单点故障风险

### 6. 管理后台模块

作用：
- 提供网页化管理入口
- 让渠道、Token、模型测试可视化

当前状态：
- 已具备最小后台雏形
- 后续会持续完善为产品级后台

## 六、技术方案概览

### 1. 后端技术栈

- **Go + Gin**：用于构建 API 服务
- **GORM**：用于数据模型与数据库操作
- **SQLite / MySQL**：用于存储配置与业务数据

这样选的原因：

- Go 部署简单
- 性能足够
- Gin 适合快速搭建 API 服务
- GORM 提高开发效率
- SQLite 适合 MVP，MySQL 适合后续正式化部署

### 2. 前端方案

当前阶段采用：
- 原生 HTML + JS 快速构建后台雏形

原因很简单：

> MVP 阶段，核心不是炫前端，而是尽快打通产品闭环。

后续如果要升级，可考虑：
- Vue3
- Element Plus
- 更完整的 SPA 管理后台

### 3. 部署方案

当前支持：
- 本地二进制运行
- Docker / docker-compose

这保证了：
- 本地测试方便
- 演示部署方便
- 后续生产化迁移也有基础

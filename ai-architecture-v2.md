## 🏗️ 架构方案

### 三层模型体系

```
                              Gateway 层
                   ┌──────────────────────────┐
                   │ primary: atmApi/qwen3.6-plus │  ← 主力
                   │ fallbacks:                │
                   │   - atmApi/qwen3.5-plus   │  ← 同路由降级
                   │   - atmApi/deepseek-v4-flash │  ← 限次兜底
                   │ safety: qwen/qwen3.5-plus │  ← 直连阿里（月套餐兜底）
                   └──────────┬───────────────┘
                              │ 全部走 atmApi 路由

                   └──────┬─────┬──────┬───────┘
                          │     │      │
              ┌───────────┘     │      └────────────┐
              ↓                 ↓                   ↓
        ┌────────────┐   ┌───────────┐   ┌────────────────┐
        │ 通义千问    │   │  通义千问  │   │   DeepSeek     │
```

## 🔧 atmApi 配置方案

### 渠道配置（已就绪）

| 渠道 | Models | Priority | 说明 |
|------|--------|----------|------|
| **通义千问（主）** | `qwen3.5-plus,qwen3.6-plus` | 10 | 月套餐，不限次 |
| **DeepSeek（备）** | `deepseek-chat,deepseek-v4-flash,deepseek-reasoner` | 5 | 按量计费，限次 |

### 需新增功能：配额管理

在 atmApi 的 `mas_tokens` 表中已有配额控制字段：

| 字段 | 用途 |
|------|------|
| `quota` | 总配额 |
| `used` | 已使用 |
| `remaining` | 剩余 |
| `concurrency_limit` | 并发控制 |
| `alert_threshold` | 告警阈值 |

**但当前缺少按时间段重置的配额（如每日/每小时配额）。**

建议 atmApi 新增功能：
1. **日配额**：每个智能体每天最多调用 N 次
2. **时配额**：每个智能体每小时最多调用 N 次
3. **配额到期自动重置**：每天 00:00 重置日配额
4. **超限熔断**：返回 `429 Quota Exceeded`，Gateway 自动 fallback

## 🛡️ 死循环防控体系（核心！）

### 三层防护

```
第一层：atmApi 配额控制（预防）
  每个智能体设日调用上限
  → 超限自动返回 429
  → Gateway fallback 到下一级模型

第二层：Gateway fallback 缓冲（降级）
  Primary 失败 → fallback 到备用模型
  各模型均受限 → 仍有直连 qwen 兜底

第三层：监控告警（发现）
  每 5 分钟检查 atmApi 请求日志
  检测异常频率 + 飞书通知
```

### 具体限流值

| 智能体 | 日调用上限 | 时调用上限 | 说明 |
|--------|-----------|-----------|------|
| 艾隆 🚀 | 3,000 | 500 | CEO + 心跳 + 模拟炒股 |
| 小龙女 🧚 | 2,000 | 400 | 自媒体 + 项目 |
| 小龙 🐉 | 2,000 | 400 | 编程 + 项目 |
| 盖茨 🎩 | 1,500 | 300 | 电商文案 + 项目 |
| 逍遥子 🧘 | 1,500 | 300 | 项目探索 |
| **总计** | **10,000** | **1,900** | 套餐 9 万/月 ≈ 3,000/天，留余量 |

### 死循环自动熔断流程

```
智能体进入死循环 → 每分钟调用 30+ 次
        ↓
第 1 分钟：触发时配额（500次/小时），正常
第 5 分钟：已调用 150 次，正常
第 10 分钟：已调用 300 次
第 20 分钟：已调用 500 次 → 时配额触顶！
        ↓
atmApi 返回 429 → Gateway 自动 fallback 到 DeepSeek
        ↓
DeepSeek 每天仅 5,000 次（日限额10元）
第 25 分钟：DeepSeek 额度用尽 → atmApi 再次返回 429
        ↓
Gateway fallback 到直连 qwen（安全兜底）
但直连 qwen 也在同一月套餐内
        ↓
第 60 分钟：qwen 套餐 5 小时限 6,000 次也触顶
        ↓
所有路全部熔断 → 智能体停摆 → 人工排查

损失上限：
  - qwen 月套餐内 = 0 额外费用（套餐烧掉了）
  - DeepSeek = 最多 10 元/天
  - 总损失：最多 10 元
  - vs 之前的 60 元 → 降低 83%
```

## 🔑 Key 管理

### 现状（痛点）

| 智能体 | 所用 Key | 存储位置 |
|--------|---------|---------|
| 5 个智能体 | 同 1 个 DeepSeek Key | 各服务器配置中 |

### 改进后

```
atmApi 统一管理所有 API Key：
  - 通义千问 Key：只存在于 atmApi 配置
  - DeepSeek Key：只存在于 atmApi 配置
  - Gateway 只连接 atmApi（1 个 Token）

换 Key 流程：
  以前：改 5 台服务器 → 重启 5 个 Gateway
  现在：atmApi 界面改 1 次 → 重启 1 个 atmApi
```

## 📝 总结

**一句话方案：atmApi 集中路由 + Three-Layer 控费 + 熔断保护**

```
正常 → atmApi → qwen3.6-plus（月套餐）
限流 → atmApi → qwen3.5-plus（同套餐）
超支 → atmApi → deepseek-v4-flash（日限10元）
宕机 → 直连 → qwen3.5-plus（保底）
```

之前烧了 60 元，方案实施后死循环场景最多损失 ≤10 元，降低 **83%**。
